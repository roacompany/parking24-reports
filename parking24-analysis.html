특수 패턴

  • Impersonation: SUPER_ADMIN이 /admin/partners에서 "포털 보기" 클릭 → x-impersonating-admin-id 헤더로 파트너 포털 진입. 별도 자산 분리.
  • Inline edit token: admin이 POST /api/admin/page-content/edit-token으로 HMAC-SHA256 5분 토큰 발급 → customer 페이지를 ?edit_mode=1&edit_token=...로 iframe 로드 → middleware 검증 → x-edit-mode + x-edit-admin 헤더로 EditableBlock contenteditable 활성화 (Layer 4 IA).
  • postMessage 이벤트: block:select / block:edit → admin parent에서 PUT /api/admin/page-content → revalidatePageContent → iframe reload.

잔여 점검 포인트

E-1 · 로그인 페이지 title 분리 minor

현재 <title>PARKING 24 - 공항 주차대행 예약</title>로 일반 사이트와 동일. "P24 Admin 로그인"으로 분리 권장 (UX/북마크 구분).

E-2 · 이메일 로그인 rate-limit 가시화

Supabase Auth 기본 rate-limit은 있지만 admin 영역은 별도 IP/계정 임계치 + 로그 영구화(partner_api_errors 유사 패턴) 검토.

E-3 · /admin/unauthorized 페이지 카피

역할 없는 사용자가 도착하는 page — 라이브 확인 못 했음(인증 후 도달). 카피 톤·되돌아가기 CTA 점검 필요.

E-4 · Impersonation 감사 로그

x-impersonating-admin-id 헤더를 모든 mutation API에서 감사 로그(누가 누구로 무엇을 했는지)로 영구화하는지 확인 필요.
