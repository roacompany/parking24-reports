1속도(성능) 핫스팟

A1 · Pagination 누락 (limit(500) 4곳)

P1MED
현재 상태 (검증됨)
src/app/api/admin/customers/route.ts:20      .limit(500);
src/app/api/admin/reservations/route.ts:28    .limit(500);
src/app/api/admin/settlements/route.ts:45     .limit(500);
src/app/api/admin/settlements/route.ts:60     .limit(500);
문제

고객·예약·정산 list가 모두 hardcoded limit(500). 광고 시작 후 row가 누적되면:

  • 500 row 풀 fetch → JSON 직렬화 + 네트워크 전송 비용 폭증
  • UI에서 500 row 한 번에 렌더링 (DOM reconciliation 비용)
  • 검색 필터가 클라이언트 JS 풀스캔 — 501번째 row는 영영 안 보임
개선안
// API: cursor 또는 offset/limit 표준화
GET /api/admin/reservations?limit=50&cursor=
GET /api/admin/customers?limit=50&offset=0

// UI: TanStack Table + 페이지네이션 컨트롤
// 또는 무한 스크롤 (admin은 desktop이라 페이지네이션 권장)

대상 파일 6개. Partner API에서 이미 검증된 after() + Promise.all 패턴 활용 가능.

A2 · reservations.created_at 인덱스 누락

P1LOW
현재 상태 (검증됨)

supabase/migrations/ 전수 grep 결과 idx_reservations_created_at이 없음. 존재하는 인덱스:

  • idx_reservations_status · checkin_date · customer_phone · reservation_number · partner_id · plan_id · coupon_code_id · payment_status
문제

거의 모든 admin list가 order by created_at desc로 정렬. 인덱스 없으면 PostgreSQL이 sequential scan → 데이터 누적 시 응답 시간 선형 증가. 광고 1년 후 100만 row면 단순 order by도 수백ms.

개선안 (Migration 031)
CREATE INDEX IF NOT EXISTS idx_reservations_created_at
  ON reservations(created_at DESC);

CREATE INDEX IF NOT EXISTS idx_reservations_partner_created
  ON reservations(partner_id, created_at DESC)
  WHERE partner_id IS NOT NULL;

CREATE INDEX IF NOT EXISTS idx_reservations_airport_created
  ON reservations(airport_id, created_at DESC);

단일 migration 파일. CONCURRENTLY 옵션은 transaction 밖이라 별도 운영 SQL로 적용 검토.

A3 · Dashboard 캐시 미사용

P2LOW
현재 상태 (검증됨)

grep -rn "unstable_cache\|revalidatePath\|revalidateTag" src/app/api/admin/ src/app/(admin)/ → 0 hits. admin 전체에 next.js cache 활용 없음.

대시보드는 매 요청마다 Promise.all 6개 쿼리 fresh fetch. 새로고침 빈번 시 DB 부하.

개선안
// src/app/api/admin/route.ts
import { unstable_cache } from "next/cache";

const getDashboardStats = unstable_cache(
  async (from: string, to: string) => {
    return await Promise.all([...]);
  },
  ["admin-dashboard-stats"],
  { revalidate: 300, tags: ["reservations"] }
);

// 예약 변경 시 revalidate
import { revalidateTag } from "next/cache";
revalidateTag("reservations");

customer 영역의 home-data.ts에 이미 unstable_cache 적용 사례 있음 — 동일 패턴 이식.

A4 · 차트 데이터 JS aggregate

P3MED
현재 상태

/api/admin/dashboard-charts/route.ts — 기간 내 예약 raw fetch 후 JS 메모리에서 4가지 aggregate(일별/공항별/이용권별/상태별).

개선안 (장기)

SQL GROUP BY + window function으로 server-aggregate. 또는 materialized view + 1시간 refresh:

CREATE MATERIALIZED VIEW mv_daily_revenue AS
SELECT date_trunc('day', created_at) AS day,
       airport_id, plan_id, status,
       count(*), sum(total_price)
FROM reservations
WHERE created_at >= now() - interval '90 days'
GROUP BY 1, 2, 3, 4;

CREATE UNIQUE INDEX ON mv_daily_revenue (day, airport_id, plan_id, status);
-- supabase scheduled function 또는 pg_cron 으로 1시간마다 REFRESH CONCURRENTLY

데이터 수십만 row 누적되기 전엔 현재 JS aggregate로 충분. 부하 모니터링 후 결정.

✅ 이미 잘 된 부분

  • Dashboard Promise.all 적용 (src/app/api/admin/route.ts:69) — Partner API에서 검증된 패턴 이미 사용
  • 관계 eager join (reservations select "*, airports(...), parking_plans(...), partners(...)") — N+1 없음
  • Vercel region icn1 — Supabase Seoul과 매칭

CREATE TABLE admin_action_logs (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),

  resource_id UUID,
  before_data JSONB,
  after_data JSONB,
  ip_address TEXT,
  user_agent TEXT,
  impersonating_admin_id UUID,  -- SUPER_ADMIN의 포털 보기 추적

);

C2 · Soft delete 도입

P4HIGH
현재 상태

DELETE는 hard delete. 복구 불가, 감사 추적 불가.

개선안

deleted_at TIMESTAMPTZ + 모든 SELECT에 WHERE deleted_at IS NULL 강제. RLS policy 갱신.

테이블 30+개 영향. ROI 모호 — 정말 필요할 때(법적 분쟁/실수 빈번) 진행.

P4MED

실수 시 봉합. service_role 직접 SQL이 빈번해지면 우선순위 상승.

5총평

현재 admin은 production-grade이지만 "광고 trigger 직전" 상태입니다. 광고 시작 후 데이터·트래픽 누적되면 다음 3가지가 가장 먼저 병목이 됩니다:

  1. Pagination 부재 — 500 row 초과 시 단순히 안 보임
  2. created_at 인덱스 — 정렬 순간 sequential scan
  3. Webhook 실패 가시성 — partner-api-errors UI에 resolved/alert 없음

Sprint 1 (4건, 1주) 완료가 광고 시작 안전선입니다. 나머지는 시작 후 측정·우선순위 조정.
