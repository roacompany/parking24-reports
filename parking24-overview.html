🏆 중요도 Top 11 (영향력 순)

각 항목은 광고 시작 후 미치는 비즈니스 영향력 기준 정렬. CRITICAL은 즉시 시스템 안정성 좌우, HIGH는 1-2주 내 발현, MED는 장기 운영 효율, LOW는 보조 개선.

1

A2 · Migration 031 — reservations 인덱스 3개

광고 후 데이터 누적되면 모든 admin list 응답 시간이 선형 증가 → 인덱스 추가로 sequential scan → Index Scan 보장.
#129 Sprint 1 Migration 031 idx_reservations_created_at · partner_created · airport_created (partial)
왜 CRITICAL? 현재 39 row 라 planner 가 Seq Scan 선택하지만 enable_seqscan=off 강제 시 Index Scan 적중 검증 완료. 데이터 누적 시 자동 전환되어 즉시 효과 발생. 인덱스 없으면 100만 row 도달 전 응답 시간 초과 가능.

기존 hardcoded limit(500) 4곳 제거. 광고 후 51번째 row부터 영영 안 보이는 문제 차단.
#130 Sprint 1 customers / reservations / settlements · count: exact · meta
왜 CRITICAL? Row 누적 = 시간 문제. 페이지네이션 없으면 운영자가 일부 데이터를 보지 못하는 게 아니라 존재 자체를 모름. 발견 시점에 이미 매출 누락 가능.

💡 시스템적 학습 (지식 자산)

1. CI gap은 build 단계에서만 발견되는 type error를 놓침

typecheck + lint + test만으로 부족. next build는 route handler export 검증, server/client boundary 검사 등 추가 가드를 함. verify:ci에 항상 포함 필요.

2. Vercel API target=production은 deployment만, alias는 별개

GitHub Actions workflow가 deployment 생성만 trigger하면 production alias로 자동 promote 안 됨. 향후 deployment build 완료 wait + alias 명시 호출 step 추가 권장.

3. Migration prod 적용 timing — local에 commit 안 된 파일은 detect 못 함

supabase db push --linked는 local supabase/migrations/ 폴더 scan. stash 상태에선 새 migration 안 detect됨. PR 머지 후 main pull → push 순서 강제.

4. Permissions map 이중 정의 drift

middleware ACL과 ADMIN_PERMISSIONS 별개 정의 — 한쪽만 추가 시 silent fail (403). 향후 단일 source 통합 또는 lint 자동 검출 필요.

5. fail-open 패턴이 mutation 안전성 보장

logger / partner-api-errors 모두 fail-open (INSERT 실패해도 mutation 자체 영향 없음). 감사·observability 인프라 추가 시 default 패턴.

6. E2E 검증은 임시 SUPER_ADMIN 자동 생성 + cleanup 패턴

Sprint 1부터 정착. 매 Sprint마다 신규 endpoint/UI를 실 prod에서 자동 검증. cleanup은 finally 블록 강제 + 잔여 0건 재확인.

7. partial success 패턴 (B2 bulk · B7 logger smoke)

일괄 작업에서 일부 row 실패 시 succeeded[]/failed[{id, error}] 분리 응답. UI는 성공 row만 자동 deselect + 실패 row 사용자 재시도 가능. Sprint 2 B2부터 Sprint 3 logger smoke까지 일관.
